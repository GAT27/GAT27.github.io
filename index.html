    • C4 Bot

    Automated Connect 4 Player

    Senior Design Project at Georgia Tech that consisted of four team members and a supervisor with the goal of creating a project to motivate students in STEM schools to pursue a career in engineering. It was decide that we would create an autonomous playing machine that would not only play Connect 4 against another person, but would also be semi-portable.

    Completed during Spring 2014.

  • These are some of the sketches I made from my designer notebook of how the final case may had looked; how the loader, feeder, and base would fold onto itself; and different ways of how the feeder would dispense pieces.

    The entire structure is divided into 3 parts: the loader (top section), the base (mid-bottom section), and the floor (flat foundation). The base is further divided into the feeder and the playing field. With the linear actuator already attained, I created a test loader to see how pieces would fall, along with a makeshift base structure made out of K'NEX to test the integrity and swivel points of the concept before continuing with stronger materials.

    I designed and created 2 models to be 3D printed. The loader had to be printed in 2 parts and attached together afterward due to its size, while the piece catcher for the feeder was made to allow drilling onto the actuator and to attach a micro servo.

    After getting the new parts for the base, I measured out the floor so that when the entire structure is folded, it would all fit within the floor.

    The circuit and PCB was designed by Andrew and testing was done by Zahiduzzaman and myself. The structure is controlled by a Raspberry Pi (model A) and the entire circuitry is powered by a 12V power outlet. While the actuator needs the 12V to operate, a voltage regulator is used to lower the voltage down to 5V to allow proper operation of the Raspberry Pi, the 2 servos, and the H-bridge motor driver. The motor driver allows the current to the actuator to be reversed when needed to allow left and right movement of the actuator. There is also a micro USB port to attach a screen and keyboard for debugging.

    In order for the C4 Bot to play, it uses a camera to poll (look at) the board and see that it is empty and 2 LEDs, green to go and red to wait, to signal the player when to start (player always goes first). When it detects an "appropriate" board change (appropriate as in there is one additional red piece added from its last state), it will see where the piece is and calculate where to put its own piece. Andrew worked on the the computer vision using OpenCV and coding threshold values for detecting red, yellow, and blue, and an array of the slots based on hard coded positions of the camera view. While there were other methods of approach and even a suggested use of K-means for detection, due to the time limit and limited processing power, this was our best approach, though if given the chance, I would of reworked this part of the code.

    The other major code of the bot is the AI, which was handled by Tray. Built in the circuit is a mode switcher to allow 3 levels of play, allowing the player to change modes in game. One of the proposed AI scored each slot it could take and select the slot with the highest score, but the algorithm was flawed and took too long to process.

    Defensive: This mode would prioritize on blocking the player on creating rows of greater than 2 pieces, otherwise, it would attempt to build its own rows.

    Aggressive: This mode would prioritize on completing its own rows, unless if the player is about to complete 4 in a row, then it will attempt to block it.

    Mixed: This mode randomly chooses playing either defensive or aggressive each turn.

    • HCI Helmet

    Concept for Human-Computer Interaction Device

    Semester long project for Computer Vision at Georgia Tech, the original idea was to create a helmet that a user would use as a sort of augmented reality, projecting an image in front of them and allowing them to control information with their hand. Unfortunately, due to the scope of the project, time constraints, and other complications, the project was scaled down to instead demonstrate individual components.

    Completed during Spring 2014

  • My main part was to code how to detect a flat surface plane using the depth map from an Xbox Kinect. No additional libraries were used and the code was done in Matlab. The entire algorithm was based on a research paper on detecting multiple planes. I simplified it to make it look for one certain sized plane, based on distance, in the center view in order for the algorithm to run in real time and to send a signal if a plane is detected or not. The 1st video is the latest code while the 2nd one is an earlier code.

    • Robot Arm

    Gesture Control Armature

    Created as a 2 part final project for Embedded System Design at Georgia Tech, the main idea is to control a robotic armature through only hand motions.

    Completed during Fall 2013

  • The robotic arm was premade with 3 micro servos, one at the base, one at the elbow, and one at the grip, with each servo sharing one power source, in this case, 4 AA batteries. Everything else is powered by the mbed, itself being powered by my laptop USB port (servos send a spike in current when started, which would overcurrent the mbed and break it, which is why they have separate power sources), which receives gestures from the sensor, send control signals the servos, and show debug information on the LCD screen.

    The Parallax Si1143 Proximity Sensor is used to detect an object in a 3D space relative to the space in front of it by using trilateration with 3 infrared LEDs, 1st one in the bottom-left, 2nd one in the top-left, and 3rd one in the bottom-right (order matters). The robot arm was originally going to be the first part of the project, but there was no library for it to work with the mbed, so instead the first part of the project was focused on creating and testing my own library for it. The library allows for baseline sensing, resets, and individual distance detection from each LEDs, though the sensor allows for other modes of operations that can be programmed in.
